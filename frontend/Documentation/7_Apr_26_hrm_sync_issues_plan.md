# Errum V2: HRM & Attendance Sync Workplan (7 Apr 2026)

This document outlines the identified inconsistencies and architectural gaps in the HRM (Human Resource Management) system, specifically regarding attendance marking, bulk actions, and payroll integration.

## 1. Identified Issues

### [CRITICAL] Bulk "Clock In All" Timestamp Gap
*   **Location**: `app/hrm/branch/page.tsx` -> `handleBulkMark`
*   **Issue**: The bulk action sends an array of entries to the backend but omits the `in_time` field. 
*   **Result**: 
    1.  `EmployeeAttendance` records are created with `in_time = NULL`.
    2.  The frontend button logic `!record?.clock_in ? 'Present' : 'Leaving'` fails because `clock_in` is null, so the button never switches to "Leaving".
    3.  Backend late fee calculations in `AttendanceController` are skipped because they depend on `!empty($entry['in_time'])`.
*   **Proposed Fix**: 
    - **Frontend**: Update `handleBulkMark` to inject `in_time: format(new Date(), 'HH:mm')` into the payload.
    - **Backend**: In `markAttendance`, if `status` is `present` or `late` and `in_time` is missing, default it to the current server time to ensure policy triggers.

### [UI/UX] Attendance Status Casing Inconsistency
*   **Location**: `services/hrmService.ts` -> `getTodayAttendance`
*   **Issue**: The service manually maps lowercase backend statuses (e.g., `present`) to Capital Case (e.g., `Present`) for UI consistency. However, some components check for lowercase while others check for Capital Case.
*   **Impact**: Brittle UI logic that breaks when new statuses are added or backend names change.
*   **Proposed Fix**: Standardize on a single case (preferably lowercase `present`, `late`, `absent`) throughout the entire data flow and only capitalize at the render layer.

### [INTEGRITY] Late Fee / Overtime Calculation Bypass
*   **Issue**: `PayrollController` calculates net pay by summing `late_fee` and `overtime_pay` columns. These columns are only populated during the `mark/update` event if timestamps are provided.
*   **Impact**: Managers can "forgive" late fees by using the bulk mark tool instead of individual check-ins, leading to financial inaccuracy.
*   **Proposed Fix**: Update `AttendanceController@markAttendance` to always run the `policyForDate` check. If a status is explicitly marked as `late` via bulk action, apply a standard "Late Penalty" even if minutes aren't tracked.

### [REDUNDANCY] Redundant Salary Locking
*   **Issue**: `AttendanceController@applyRewardFineToSalary` and `PayrollController@payMonthlySalary` both handle the "locking" of adjustments. 
*   **Impact**: Confusion in the UI about whether a month is "Locked" or "Paid".
*   **Proposed Fix**: Deprecate the manual `applyRewardFineToSalary` endpoint. Moving forward, the `payMonthlySalary` flow should be the sole authority for marking rewards, fines, and attendance as `is_applied = true`.

## 2. Detailed Implementation Plan

### Phase 1: Backend Robustness (Laravel)
1.  **Standardize Statuses**: Update `EmployeeAttendance` model and migration (if possible) or just the controller to strictly use lowercase strings.
2.  **Auto-Timestamping**: Modify `AttendanceController@markAttendance`:
    - If `in_time` is null but status is `present`/`late`, set `in_time` to `now()`.
    - If `out_time` is null but status is `leaving`, set `out_time` to `now()`.
3.  **Policy Enforcement**: Ensure `late_fee` calculation runs even if `in_time` was generated by the server.

### Phase 2: Frontend Sync (Next.js)
1.  **Fix Bulk Action**: Update `handleBulkMark` in `app/hrm/branch/page.tsx` to include timestamps.
2.  **Button Logic Refactor**: 
    - Update the button condition in `Staff Attendance` table:
      ```typescript
      const isClockedIn = record && (record.clock_in || record.status === 'Present' || record.status === 'Late');
      const isClockedOut = record && record.clock_out;
      ```
3.  **Service Normalization**: Remove manual string mapping in `hrmService.ts` and use a utility function for display labels.

### Phase 3: Payroll & Reporting
1.  **Consolidate Adjustment Logic**: Move the `is_applied` update logic entirely into the `payMonthlySalary` transaction in `PayrollController`.
2.  **Audit Logs**: Ensure that "Bulk Mark" actions are recorded in `EmployeeAttendanceHistory` with a flag indicating it was a manager-led bulk action.
